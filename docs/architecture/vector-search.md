# Rowset Vector Search Architecture

## Scope

Rowset vector search indexes ready dataset rows for authenticated agent and REST
search. Rowset/Postgres remains canonical. Qdrant stores a rebuildable retrieval
index and must not become the source of truth for rows, dataset metadata, ACLs,
or public preview state.

V1 supports dataset-level row search. Cross-dataset, public-preview, and
workspace-wide knowledge search are out of scope until dataset search is stable.

## Retrieval Index Role

Qdrant stores one point per Rowset row chunk. V1 emits one chunk per structured
row. Future document-style datasets can add more chunks without changing the row
hydration contract.

Each point stores:

- a deterministic point id derived from dataset key, row id, and chunk index
- a named dense vector generated by the configured embedding model
- metadata needed for server-derived filtering and Rowset hydration

Large canonical objects stay in Postgres and dataset asset storage. Search
results hydrate rows from Rowset after retrieval, then serialize through the
same row serializer as normal API and MCP row reads.

## Collection Strategy

Collections are grouped by application, content kind, embedding model,
dimensions, and collection version:

```text
{prefix}_rows_{embedding_model}_d{dimensions}_v{version}
```

The default prefix is `rowset`. Do not mix incompatible embedding dimensions in
one collection. A model or dimension change should write to a different
collection, then backfill from Rowset.

## Payload Schema

Dataset row points include these payload fields:

- `app`: `rowset`
- `content_type`: `dataset_row`
- `profile_id`
- `dataset_id`
- `dataset_key`
- `dataset_status`
- `dataset_archived`
- `row_id`
- `row_number`
- `index_column`
- `index_value`
- `chunk_index`
- `content_hash`
- `embedding_model`
- `embedding_dimensions`

The server derives required filters. Clients cannot choose tenant or dataset ACL
filters directly.

## Indexing Flow

Rowset indexes rows after canonical writes commit:

1. Dataset creation with initial rows enqueues a dataset backfill.
2. Row create/update enqueues one row indexing task.
3. The worker fetches the canonical row and dataset.
4. The worker renders deterministic search text from the dataset name, index
   column, headers, column descriptions, and row values.
5. The embedding provider returns a vector with model and dimensions metadata.
6. Qdrant upserts the point into the configured collection.

Indexing failures are logged by workers and do not roll back Rowset writes.

## Backfill And Reindex

Operators can backfill an existing ready dataset with:

```bash
python manage.py backfill_dataset_vectors <dataset_key> --dry-run
python manage.py backfill_dataset_vectors <dataset_key>
```

The command streams rows in row order, batches embedding requests and Qdrant
upserts, and reports rows seen, indexed, and failed. A model or dimension change
should update collection naming through settings, run a scoped backfill, and
switch search to the new configured collection.

## Deletion And Archival

Row deletes enqueue Qdrant cleanup for the deleted row id. Bulk row deletes
enqueue cleanup for all deleted row ids. Dataset archive enqueues cleanup for
all points in that dataset.

Search also hydrates from the authenticated active dataset. If a stale point
survives temporarily, a missing or inaccessible Rowset row is skipped instead of
returned.

## Query Flow

1. REST or MCP authenticates the caller and resolves one ready dataset owned by
   the caller.
2. Rowset embeds the query with the configured embedding provider.
3. Qdrant receives dense vector search with server-derived payload filters for
   app, content type, profile, dataset, ready status, and non-archived state.
4. Rowset runs canonical lexical row search and row filters in Postgres.
5. Rowset combines vector and lexical candidates with reciprocal-rank fusion.
6. Rowset hydrates ranked row ids from Postgres and serializes rows plus match
   metadata.

The response includes rank, score, source, vector rank, lexical rank, vector
score, point id, chunk index, content hash, snippet, and the hydrated row.

## Observability

Search emits one structured completion log per successful request. The log uses
a generated `query_id` and includes dataset/profile identifiers, limit, filter
count, vector hit count, lexical candidate count, fused candidate count, result
count, hydration misses, top-result source and scores, embedding model,
embedding dimensions, embedding latency, Qdrant search latency, and total search
latency.

Worker logs cover row indexing completion/failure, dataset backfill
completion/failure, row vector deletion completion/failure, and dataset vector
deletion completion/failure. Backfill completion includes rows seen, indexed,
and failed.

Quality regressions are covered by a small deterministic test fixture with
taskboard and CRM-style rows. It checks exact identifiers such as
`ROW-VEC-001`, fuzzy semantic queries such as stale-vector cleanup, quality
fixture queries, and CRM renewal-risk lookups.

## Operational Notes

Keep vector debug output free of raw private row text, embeddings, API keys, and
provider secrets. Log counts, dataset keys, provider failures, Qdrant failures,
and hydration misses instead.

Qdrant can be rebuilt from Rowset/Postgres. If the index is suspected stale or a
model rollout fails, disable vector search, fix configuration, and rerun scoped
backfills.
